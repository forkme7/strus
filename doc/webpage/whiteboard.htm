Strus whiteboard

The Strus whiteboard is a page for brainstorming. Here we put our ideas and draw the first sketches of the near future of strus. You are welcome to work with us on the whiteboard.
Planned articles on codeproject
(SHORT-TERM)
The following articles on codeproject are planned for the next weeks:
- Show the expandability of the Strus core. Explain how to build modules for it. Introduce the interfaces for weighting functions, summarizers, expression joins. Do this also for the analyzer. Introduce the interfaces for segmenters, tokenizers, normalizers, aggregators.
UIMA integration
(MEDIUM-TERM)
The Strus analyzer might be a good choice for small projects. It does not have too many dependencies. It has in my opinion some expressive languages to describe document transformation. It is very efficient.
But it has limits, there is a wall. Document processing is an extremely competitive environment with many mature solutions available. You cannot compete there without a strong focus on it and many man years of valuable development behind you.
The Strus analyzer has also been developed as separated project, because this is not the ground to fight for Strus.
One framework that looks very interesting is UIMA. It would be interesting to build a document processing chain with UIMA for feeding Strus.
Cooperation
(MEDIUM-TERM)
Strus has already reached the limits, where it is healthy for the project to be dependent on only one or two brains. Sharing ideas is the primary stimulation for development. A first step could be to find open source projects that probably need a search and try to integrate there.
For people out there reading these lines: Please mail us (contact (at) project (dash) strus (dot) net) if you have problems, ideas or you are looking for cooperation.
Introspection of Strus objects on client side
(LONG-TERM)
Something missing in strus is introspection of the data structures built in a program (expression trees, function definitions, data loaded and processed etc.).
Because this seems to be an aspect on its own, we could implement an own proxy universe from StorageObjectBuilderInterface and AnalyzerObjectBuilderInterface, similarly to what strusModule (the module loader) or strusRpc does. Every object created would implement its method calls as redirect to the real method call and additionaly build some representation in a universe that is introspectable (an object tree).
